What is Version Control?

Version Control Defined

Version Control (also known as Revision Control, Source Control, Source Control Management, Source Code Management) by definition is the management of changes between documents

The History of Version Control

  • The simplest form of a version control system is duplicating all your documents and renaming the backups with unique revision numbers. The next step would be to use a comparison tool or manually search for changes between backups and combine the differences into a new backup.
  • Luckily, version control systems have evolved into automated command-line based programs that achieve the same level of management over your changes and more.

Types of Version Control Programs

Centralized

Examples of Centralized Programs
Subversion (SVN), Bazaar, Version Cue, Perforce
All changes are tracked (though commits) in a server-side remote repository
The user or client arbitrarily checks out and commits changes to the remote repo. Then the remote repo handles the changesets (called deltas), merging and all other version control operations.

Distributed

Examples of Distributed Programs
Git, Mercurial, darcs
EVERYONE (both client and server) has a copy of the full repository, meaning they have full access to the entire history of the repo including every commit ever made.
All repo operations like handling changesets, merging, etc. occur locally.

Why Git?

Why Git?

Git is Distributed
Everyone has a full copy of the repo
If a remote server crashes, everyone still has the full copy locally and no history is lost.
You aren't dependent on an Internet connection so your normal workflow for committing changesets isn't interrupted.
Having the full repo locally promotes more frequent and specific commits of changesets. This gives a more granular and accurate history of all the changes throughout your project.
Since most repo operations (like commit, merge, diff, checkout, etc) are handled locally, this dramatically minimizes the amount of bandwidth usage needed.
Git has a Stage
Well, what is a stage? We'll get to that in a second.
Git is Fast
Git executes commands much faster than other systems like Subversion because Git handles the majority of its operations locally. Subversion and Mercurial require network access for most network operations

The Basic Git Workflow

  • In Subversion, the repo and changesets (commits) are stored on the remote server. The user checks out the most recent changeset to their local working environment.
  • In Git, the user has a full copy of the repo and changesets (commits) locally. All of the operations for the basic Gt workflow can be accessed without Internet connection.
  • Git has the addition of a "staging" area (or index). The staging area allows you to succinctly craft your commits. For example, if you work for 3 hours on several tasks (like adding a new feature and fixing a QA bug), in Subversion you would have to commit all of these changes together to the server in a single gigantic commit. However, if you use Git, you can add specific files to the stage and then commit these changes individually as separate tasks, rather than one giant unorganized commit. This makes for a more accurate and granular history that describes exactly what code changes map to what tasks.

The Basic Git Workflow in Action

Initializing Git
$ git init
Adding a file or making code edits through the text editor of your choice
Checking the status of your repo
$ git status
Adding the unstaged changes
$ git add [path]
eg. $ git add css/master.css
Committing your staged change to your local repo with a descriptive message
$ git commit -m [your message]
eg. $ git commit -m "Fixed bug #4 by adding a hasLayout fix to the sidebar div"
Finding Help
$ git help
eg. $ git help bisect (shift + q to escape)

The Basic Git Workflow with Remote Repos

  • There is nothing special about remote repos. They are an exact copy of what you have locally.
  • Anything running Git on their server could be called a remote repo. The common practice when you want a remote repo is to use a web service that has Git running on their servers so you can host your code remotely and most provide a web-based interface for visualizing the history and activity of the remote repo. These web services often bundle issue tracking and wikis as well.

List of Remote Repo Web Services

  • GitHub
  • Unfuddle

The Basic Git Workflow with Remote Repos in Action

Once you're ready to host your code, Add a remote repo
$ git remote add [remote name] [remote url]
eg. $ git remote add origin git@codebasehq.com:cmdagency/my-awesome-new-remote-repo/
Follow the Basic Git Workflow by making code edits, add and commit all changes to your local repo
Pull any new changes from the remote repo to your local repo
$ git pull [remote name] [branch name]
eg. $ git pull origin master
Push your local changes to the remote repo
$ git push [remote name] [branch name]
eg. $ git push origin master

A Little Bit More About Working with Remote Repos

  • Again, anything running Git could be a remote repo.
  • The workflow for working with remotes can be centralized and tailored around one main remote repo, but it doesn't have to be nor is the Git workflow intended to be.
